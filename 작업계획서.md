## 0. 이 문서를 읽는 법

- **§1~§2**: 무엇을/왜 만드는가 (범위·목표·성공 기준).
- **§3 (ADR)**: 초안 대비 바뀐 핵심 결정과 그 근거. **여기가 이 문서의 핵심 개선점**이다.
- **§4~§7**: 어떻게 만드는가 (스택·아키텍처·데이터·모노레포 구조).

각 단계(Phase)에는 **산출물(Deliverables)** 과 **완료조건(Definition of Done, 검증 가능한 형태)** 를 명시했다.

## 1. 프로젝트 개요

- **목적**: gRPC 고성능 네트워크 통신을 활용한 실시간 대용량 영상 스트리밍 및 라이브 채팅 플랫폼 구현.
- **핵심 모델**: 트위치(Twitch)의 핵심 UI/UX 및 기능 벤치마킹.
- **주요 특징**: 마이크로서비스 간 gRPC 통신 최적화, 실시간 데이터 브로드캐스팅, 확장 가능한 아키텍처.

### 1.1 범위 (In Scope) — MVP 기준

| #   | 기능           | 설명                                                        |
| --- | -------------- | ----------------------------------------------------------- |
| F1  | 계정/인증      | 회원가입·로그인(JWT access+refresh), 프로필                 |
| F2  | 채널/방송 관리 | 채널 생성, 방송 시작/종료, 스트림 키 발급·검증              |
| F3  | 방송 목록/탐색 | 라이브 목록, 카테고리, 실시간 시청자 수                     |
| F4  | 영상 시청      | RTMP 인제스트 → LL-HLS 재생 (hls.js)                        |
| F5  | 실시간 채팅    | 채널별 채팅방, 브로드캐스트, 기본 모더레이션(밴/슬로우모드) |
| F6  | 팔로우/알림    | 채널 팔로우, 라이브 시작 알림(간단)                         |

### 1.2 비범위 (Out of Scope) — MVP 이후

- 수익화(구독/도네이션/광고), 클립·VOD 편집, 추천 알고리즘, 모바일 네이티브 앱.
- E2E 초저지연 WebRTC 대규모 배포(§3 ADR-3에서 "선택지"로만 남김).
- **ABR(적응형 비트레이트, 다중 화질 트랜스코딩)** — MVP는 단일 화질(패스스루/단일 렌디션)만. 다중 렌디션은 후속. _(리뷰 반영: 묵시적 in-scope 방지)_
- **풀 관측성 스택(OTel 분산추적 + Grafana + Loki)** — MVP는 pino 구조화 로그 + 기본 Prometheus 메트릭까지만. 분산추적/대시보드/중앙로그는 후속(§10). _(리뷰 반영: 8주 범위 과대 방지)_
- **i18n(다국어)** — MVP는 단일 언어(ko). 구조만 열어두고 번역 리소스는 후속.
- **채팅 로그 영구 아카이브** — MVP는 Redis 실시간만, 영구 저장/검색 없음(개인정보/GDPR 표면 최소화, §10).

### 1.3 성공 기준 (측정 가능)

> **측정 전제**: 아래 성능 수치는 **§8.0 환경 정의**의 "부하 측정 환경(sized host)"에서 지정 방법으로 측정한 값이며,
> 개발자 노트북 docker-compose 값이 아니다. 각 수치는 **pass/fail 게이트**이며, 미달 시 "수용된 편차(accepted deviation)"로 근거와 함께 문서화해야 통과로 간주한다. _(리뷰 반영: 측정 가능성/게이트화)_

- **기능(최우선)**: §1.1 F1~F6 시나리오가 **실제 배포된 환경에서 사람이/스모크 스크립트로 동작 확인**(pass/fail). 자동 E2E는 **해피패스 1개(로그인→목록→시청→채팅)** 만 필수, 나머지는 수동/스모크로 대체.
- **배포(이번 목표)**: `main` 머지 시 **web은 Vercel, 백엔드·미디어는 Fly.io/Railway로 자동 배포**되고, 배포 후 `/health`·재생·채팅 스모크가 **실 도메인에서 통과**(pass/fail). ADR-8·Phase 6 참조.
- **성능(부하 측정 환경 기준)**:
  - 방송 목록 API p95 < 100ms (Redis 캐시 히트 시) — k6 HTTP.
  - 채팅 메시지 팬아웃 지연(발신→타 시청자 수신) p95 < 300ms @ 1,000 동시 접속/채널 — **k6 WebSocket 부하 리그(§8.0)** 로 측정.
  - 재생 시작 지연(플레이어 로드→첫 프레임) < 3s (LL-HLS), **glass-to-glass** 지연 < 6s — **송출 화면 타임코드(가시 타임스탬프) 프레임 대조법**으로 측정(측정 방법을 Phase 3 산출물로 문서화).
- **품질(테스트 최소화 원칙)**: 커버리지 수치 목표는 **두지 않는다**. 대신 **크리티컬 패스(인증·스트림키 검증·팬아웃·재생 인가)만 얇은 스모크/단위 테스트로 회귀 방어**하고, `tsc(strict)`·`buf breaking`·CI 그린을 계약 안전망으로 삼는다. _(v1.2: 테스트보다 실제 구현·배포 우선)_

### 1.4 팀 구성 전제 & 일정 조건 _(리뷰 반영: 무근거 추정 제거)_

이 계획의 **8주 일정은 다음 전제에서만 성립**한다.

- **인원**: 약 4~~5명(프론트 1~~2, 백엔드 2, **미디어 전담 1**), TypeScript/gRPC 경험 보유.
- **조건**: (a) ABR·풀 관측성(Grafana/Loki/OTel) 후속 이월, (b) **미디어 스파이크를 Phase 0/1로 선행**(§8, R1), (c) Phase 3를 미디어 트랙과 프론트-실시간 트랙으로 분리·병렬화.
- **현실 보정**: 인원 1~2명이거나 미디어 스파이크를 Phase 3에 남기면 **실질 12~20주**로 본다. 일정 압박은 R7에서 상향 관리(§11).

## 2. 아키텍처 전략 요약

초안의 3-서비스 분리를 유지하되, 브라우저 gRPC 제약을 **BFF + Connect 프로토콜**로 정면 해결한다.

```
                         ┌──────────────────────────────────────────┐
   [ OBS / 인코더 ]      │              내부망 (gRPC over HTTP/2)      │
        │ RTMP           │                                            │
        ▼                │   ┌───────────────┐   ┌───────────────┐    │
 [ 미디어 인제스트 ]──────┼──►│ Core(방송/유저) │◄─►│ Chat(채팅)     │    │
   RTMP→LL-HLS 패키징     │   └───────┬───────┘   └───────┬───────┘    │
        │ .m3u8/.m4s      │           │                   │            │
        ▼                │           ▼                   ▼            │
 [ Object Storage/CDN ]  │      [ PostgreSQL ]        [ Redis ]       │
        ▲                └───────────┼───────────────────┼───────────┘
        │ HLS 재생                    │  gRPC(Connect)     │ Pub/Sub
        │                    ┌────────┴───────────────────┴────────┐
        │                    │   BFF / API Gateway (Connect-Node)   │
        │                    │  · REST/Connect(HTTP)  · WS(채팅)     │
        │                    └───────────────────┬──────────────────┘
        │                                        │ Connect-Web / WS / HLS
   [ 브라우저 (Next.js + hls.js) ]◄───────────────┘
```

- **제어/데이터 평면 분리**: API·채팅(제어/텍스트)와 미디어(영상 바이트)는 완전히 다른 경로로 흐른다. 영상은 절대 gRPC로 나르지 않는다(§3 ADR-3).
- **BFF는 유일한 _제어 평면_ 진입점**: 브라우저의 모든 API/RPC/채팅은 BFF 경유(Connect-Web/REST/WS), BFF가 내부 gRPC로 중계. **단, 미디어 데이터 평면(HLS 세그먼트)은 의도적으로 BFF를 우회**해 CDN/스토리지에서 직접 fetch한다. 이때 재생 인가는 §5.2의 **서명된 단기 HLS URL**로 처리한다. _(리뷰 반영: "유일 진입점" 표현이 미디어 우회와 모순되던 점 수정)_

## 3. 아키텍처 결정 기록 (ADR) — 초안 대비 변경점과 근거

> 초안이 "이 중 하나를 골라라"로 남긴 지점들을 **결정 + 근거 + 대안 + 트레이드오프**로 확정한다.

### ADR-1. 브라우저 통신: gRPC-Web 대신 **Connect (connectrpc)** 채택 ✅

- **결정**: 프론트–BFF 간 RPC는 **Connect 프로토콜**(`@connectrpc/connect-web`, 백엔드 `@connectrpc/connect-node`)로 통일. 채팅 실시간은 WebSocket(또는 Connect 스트리밍) 사용.
- **근거**:
  - 전통적 gRPC-Web은 **Envoy/NGINX gRPC-Web 필터**라는 별도 프록시가 필수 → 운영 복잡도 급증(초안 §5의 "정신 건강" 경고 지점).
  - Connect는 **동일 `.proto`에서 브라우저·서버 코드를 함께 생성**하고 HTTP/1.1·HTTP/2·gRPC·gRPC-Web·Connect 프로토콜과 **와이어 호환**된다. 별도 프록시 없이 브라우저에서 바로 호출 가능.
  - 내부 서비스 간에는 그대로 표준 gRPC(HTTP/2)로 통신 → "내부 gRPC 최적화"라는 초안 목표를 훼손하지 않음.
- **대안**: (a) 순수 gRPC-Web + Envoy — 검증되었으나 프록시 운영 부담. (b) tRPC — DX는 좋으나 다중 언어/proto 자산과 무관, gRPC 학습 목표와 어긋남.
- **브라우저 스트리밍 제약(중요)**: `@connectrpc/connect-web`의 fetch 트랜스포트는 **unary + 서버 스트리밍만** 지원한다. **클라이언트 스트리밍/양방향(bidi)은 브라우저에서 불가**(브라우저가 요청 바디를 버퍼링하는 플랫폼 한계). 따라서 실시간 채팅처럼 클라이언트→서버 지속 스트림이 필요한 경로는 **WebSocket으로 처리**(ADR-4)한다. 서버→클라이언트 단방향 푸시(예: 알림)는 Connect 서버 스트리밍으로 가능.
- **트레이드오프**: Connect 생태계 학습 필요. 단, proto/코드젠 자산은 표준 gRPC와 공유되어 락인 위험 낮음.

### ADR-2. Proto 거버넌스: **buf** 도입 ✅ (초안에 없던 항목)

- **결정**: `.proto` 관리는 **buf**로 표준화 — `buf lint`, `buf format`, `buf breaking`(호환성 파괴 감지), `buf generate`(코드젠). 코드젠 플러그인은 **`protoc-gen-es` 단독**을 사용한다.
- **버전 주의(중요)**: **Connect-ES 2.0(현행)** 부터 `protoc-gen-es` 하나가 Protobuf 메시지 + Connect 서비스 코드를 함께 생성하므로 **`protoc-gen-connect-es`는 더 이상 사용하지 않는다**(1.x 유물). `buf.gen.yaml` 예:
  ```yaml
  version: v2
  plugins:
    - local: protoc-gen-es
      opt: target=ts
      out: src/gen
  ```
  _(리뷰 반영: 초안의 2-플러그인 구성은 Connect-ES 1.x 기준으로 outdated)_
- **근거**: proto는 서비스 계약. 손으로 `protoc` 관리 시 옵션·경로 지옥. buf가 lint/breaking-change/코드젠을 일원화하고 CI에 물리기 쉽다.
- **트레이드오프**: 초기 설정 학습. 이후 계약 변경 안전성 대폭 상승.

### ADR-3. 미디어 경로: **RTMP 인제스트 → LL-HLS 패키징** (기본), WebRTC는 선택 ✅

- **결정(MVP)**: 스트리머는 OBS 등으로 **RTMP** 송출 → 미디어 인제스트 서버가 **LL-HLS**(Low-Latency HLS)로 패키징 → 오브젝트 스토리지/CDN 경유 → 브라우저 `hls.js` 재생.
- **근거**: RTMP 인제스트는 OBS 등 기존 도구와 호환, HLS는 CDN 확장이 쉽고 브라우저 호환성이 넓다. LL-HLS로 지연을 실용 수준(수 초)까지 낮춘다. **영상 바이트는 gRPC로 나르지 않는다** — gRPC는 제어·메타데이터·채팅에만.
- **대안/향후**: 초저지연(<1s)이 필요하면 **WebRTC SFU**를 별도 트랙으로 추가. SFU 선택지는 `mediasoup`(저수준 제어·자체호스팅 역량 필요) vs **LiveKit**(2026 신규 프로젝트 기본 권장, 운영 부담↓). MVP에서는 복잡도/비용 때문에 제외하고 인터페이스만 열어둔다.
- **구현 후보 & 검증 필수(중요)**: 인제스트/패키징 1순위는 `Node-Media-Server`(단순, 학습용) 또는 `FFmpeg` 파이프라인 직접 구성.
  - ⚠️ **`Node-Media-Server`의 진짜 LL-HLS(부분세그먼트 `#EXT-X-PART`/`#EXT-X-PRELOAD-HINT`, 블로킹 재생목록 리로드) 지원은 공식 문서상 미확인**이다. §1.3의 지연 목표(재생시작 <3s, g2g <6s)가 실제 LL-HLS에 의존한다면 NMS로는 커스텀 없이 도달 못 할 수 있다.
  - 따라서 **미디어 스파이크(§8, Phase 0/1로 선행)에서 NMS의 LL-HLS 실측을 검증**하고, 미달 시 **MediaMTX / SRS로 전환**한다. "향후 검토"가 아니라 **Phase 3 착수 전 필수 결정 게이트**로 승격. _(리뷰 반영: 미디어가 일정 최대 리스크)_
- **썸네일/미리보기 & 세그먼트 수명주기**: 방송 목록 카드용 **주기적 썸네일 캡처**(FFmpeg 스냅샷)와, HLS 세그먼트/썸네일의 **스토리지 TTL 정리(retention)** 를 MVP 범위에 포함(§8 Phase 3). _(리뷰 반영: 목록 UX·스토리지 증가 대응)_

### ADR-4. 실시간 채팅 전송: **WebSocket(주) + Redis Pub/Sub(팬아웃)** ✅

- **결정**: 브라우저↔BFF 채팅은 **WebSocket**(`ws` 또는 Socket.IO). 서버 인스턴스 간 팬아웃은 **Redis Pub/Sub**. 채팅 서비스 로직은 gRPC로 Core와 통신(밴 여부/권한 조회 등).
- **근거**: 채팅은 다수 시청자에게 **낮은 지연 브로드캐스트**가 핵심. WS는 브라우저 네이티브 지원 + 양방향. Connect 스트리밍도 가능하나, 대규모 팬아웃/재연결/룸 관리 성숙도는 WS 생태계가 우위.
- **트레이드오프**: WS와 Connect 두 전송 방식을 함께 운용. 대신 각자 강점 영역에 배치.
- **팬아웃 정합성(중요)**: Redis Pub/Sub는 모든 BFF 인스턴스에 브로드캐스트되므로 **전달을 위한 sticky session은 불필요**(§10의 "룸 샤딩"은 정합성이 아니라 확장 최적화). 단 인그레스 측을 명시한다:
  - **BFF WS 계층은 룸별 refcount 구독** — 해당 룸 첫 조인 시 Redis 채널 구독, 마지막 이탈 시 구독 해제(소켓마다/전역 구독하는 나이브 구현 금지).
  - **역할 분리**: BFF = WS 종단 + 릴레이만. **Chat 서비스 = 모더레이션/슬로우모드/권한 판정 + (선택)영속 + Redis publish**. 즉 Chat은 핫패스의 정책·발행 담당이지 불필요한 홉이 아니다. _(리뷰 반영: 구독 수명주기·책임 분리 명시)_
- **확장 주의**: Pub/Sub는 at-most-once. 히스토리/재전송이 필요하면 **Redis Streams**로 승격(§13 확장성).

### ADR-5. 프레임워크: 백엔드 **NestJS**, 프론트 **Next.js(App Router)** ✅ (초안 권고 확정)

- **근거**: NestJS는 gRPC 마이크로서비스·DI·모듈 경계를 1급 지원 → 3-서비스 구조에 적합. Next.js는 방송 목록 SEO(SSR/ISR)와 Connect-Web 연동, 이미지/스트리밍 UX에 유리.
- **트레이드오프**: NestJS 추상화 학습. 순수 gRPC 최소구성 대비 무겁지만 팀 생산성·구조화 이득이 큼.

### ADR-6. 데이터 계층: **PostgreSQL + Drizzle ORM**, **Redis**, **Object Storage(S3/MinIO)** ✅

- **결정**: 관계형 영속 데이터는 **PostgreSQL**(초안의 "PostgreSQL 또는 MongoDB" → 관계·트랜잭션이 명확한 도메인이므로 Postgres 확정). ORM은 **Drizzle**(타입세이프·경량·마이그레이션). 실시간 상태·캐시·Pub/Sub은 **Redis**. VOD/썸네일/HLS 세그먼트는 **오브젝트 스토리지**(로컬은 MinIO, 클라우드는 S3 호환).
- **대안**: ORM으로 Prisma(성숙, 무겁고 엔진 바이너리 의존) — 팀 선호 시 대체 가능. MongoDB는 채팅 로그 아카이브 등 특정 용도로만 선택적.

### ADR-7. 모노레포: **pnpm workspaces + Turborepo** ✅ (초안에 없던 항목)

- **근거**: 백엔드 3서비스 + BFF + 프론트 + 공유 proto/스키마/타입을 한 저장소에서 버전 정합성 있게 관리. Turborepo로 빌드 캐시/태스크 파이프라인 최적화.

### ADR-8. 배포 전략: **하이브리드 (Vercel[web] + Fly.io/Railway[백엔드·미디어] + R2/S3[HLS CDN])** ✅ (v1.2 신설, 이번 프로젝트 목표)

- **결정**:
  - **프론트(`apps/web`, Next.js)** → **Vercel**. SSR/ISR·프리뷰 URL·CDN 이점을 그대로 활용. 브라우저는 Connect-Web/WS로 BFF의 **퍼블릭 도메인**을 호출(크로스오리진 → §10 CORS/WS 규약).
  - **백엔드(`bff`, `svc-core`, `svc-chat`) + Postgres + Redis** → **Fly.io**(1순위) 또는 Railway. 컨테이너 상시 실행, 내부 gRPC는 사설 네트워크(Fly `.internal` / Railway private networking)로 통신. Postgres/Redis는 매니지드(Fly Postgres·Upstash Redis) 또는 컨테이너.
  - **미디어(`svc-media`, RTMP→LL-HLS)** → **Fly.io**(RTMP은 **raw TCP 1935** 필요 → Fly `[[services]] protocol="tcp"` + 전용 IP). Railway는 TCP 프록시 제약이 커 **미디어는 Fly 우선**.
  - **HLS 세그먼트·썸네일 CDN** → **Cloudflare R2**(S3 호환, egress 무료가 비디오 특성에 유리) 또는 S3+CloudFront. 로컬(dev)은 MinIO 유지 → **S3 호환 인터페이스로 dev/prod 코드 동일**.
- **근거**: 서버리스 단독이 불가한 스택(RTMP 상시 TCP, 상태 gRPC, WS 롱커넥션, Redis Pub/Sub)을 컨테이너 호스트로 해결하되, 프론트만은 Vercel의 SSR/CDN/프리뷰 DX를 취하는 실용적 분리. R2는 비디오 egress 비용의 핵심 리스크(§10)를 낮춘다.
- **트레이드오프**: (a) **경계가 두 플랫폼에 걸쳐 CORS·WS 오리진·서명 URL 도메인 설정이 필수**(§10). (b) 컨테이너 콜드스타트/스케일-투-제로 시 첫 요청 지연 → 미디어·BFF는 **min-instance ≥ 1**로 상시 가동. (c) 시크릿이 Vercel·Fly 양쪽에 분산 → §10 시크릿 관리 규약.
- **대안**: 단일 VPS+Compose(운영 단순하나 프리뷰/오토스케일 없음), k8s(과함), 풀 매니지드 AWS(비용·IaC 부담). 본 프로젝트는 학습·MVP·데모 목적상 하이브리드가 균형점.

## 5. 시스템 아키텍처 상세

### 5.1 서비스 경계

1. **Core 서비스 (gRPC)**: 유저/인증, 채널, 방송 세션(시작/종료), 스트림 키 발급·검증, 방송 목록/카테고리. **라이브 상태(`streams.status` + `live:{channelId}`)의 유일 writer**(ADR-3). 소유 데이터: Postgres(users, channels, streams, follows).
2. **Chat 서비스 (gRPC + WS)**: 채팅 룸, 메시지 브로드캐스트, 슬로우모드/밴 적용, **시청자 수 집계**. 시청자 수는 Chat이 Redis `viewers:{channelId}`에 델타로 반영하고, **Core가 `ListLive` 조립 시 이 키를 읽는다**(집계 소유=Chat, 노출=Core). 소유 데이터: Redis(pub/sub, 룸 상태, viewers). _(리뷰 반영: 시청자 수 소유 경계 명시)_
3. **Media 인제스트/패키저**: RTMP 수신 → 스트림 키를 Core에 gRPC로 검증 → LL-HLS 패키징 → 스토리지/CDN. 방송 시작/종료 이벤트를 Core에 통지.
4. **BFF / API Gateway**: 외부 유일 _제어 평면_ 진입점(미디어 데이터 평면은 §5.2 서명 URL로 우회). 브라우저 ↔ (Connect-Web/REST/WS), 내부 ↔ (gRPC). 인증 토큰 검증, 요청 조립(aggregation), WS 채팅 종단.

### 5.2 핵심 데이터 흐름

- **방송 시작(스트리머)**: OBS(RTMP, `rtmp://.../live/<streamKey>`) → Media 인제스트 → Core에 `StartStream`(gRPC, 내부에서 `ValidateStreamKey`) 호출. **라이브 상태의 실제 쓰기는 Core만 수행**: Core가 `streams.status` + `live:{channelId}`를 원자적으로 기록(Media는 통지만, 직접 Redis에 쓰지 않음). 라이브 상태 무결성 3대 규칙:
  - **중복 송출 방지**: 세션 오픈은 Redis `SET NX`로 획득 → 같은 키로 들어온 2번째 인코더는 거부.
  - **좀비 스트림 방지**: `live:{channelId}`에 **TTL** + Media의 **주기적 heartbeat RPC** → 랩스 시 Core가 세션 회수(ended 처리).
  - **재생 준비 완료 후 노출**: **첫 세그먼트 기록 확인 후에만** `live`로 노출(또는 플레이어 재시도 정책 명시) → 초기 시청자 404 방지. _(리뷰 반영: 단일 writer·TTL/heartbeat·SET NX·타이밍)_
- **시청(뷰어)**: 브라우저 → BFF `GetChannel`/`GetPlaybackUrl`(Connect) → **서명된 단기 HLS URL**(CDN signed cookie 또는 토큰화 `.m3u8`, 짧은 TTL) 수신 → `hls.js`가 CDN에서 세그먼트 직접 fetch(BFF 우회, 데이터 평면). **재생 인가는 이 서명 URL로 강제**(오픈 URL 무단 시청 차단). _(리뷰 반영: 데이터 평면 재생 인가 부재 보완)_
- **채팅**: 브라우저 WS 연결 → BFF가 인증·룸 조인 → 메시지 수신 시 Chat 서비스로 전달 → Redis Pub/Sub 팬아웃 → 각 BFF 인스턴스가 자기 소켓들에 push.
- **목록**: 브라우저 → BFF `ListLiveChannels`(Connect) → Core가 Redis 캐시(활성 방송) 조회, 미스 시 Postgres → 응답.

### 5.3 인증·인가 흐름

- 로그인 → Core가 argon2 검증 → access(JWT, 단명) + refresh(회전형, httpOnly 쿠키) 발급.
- BFF가 매 요청 access 검증(공유 시크릿/JWKS). 만료 시 refresh로 재발급.
- gRPC 내부 호출은 mTLS(🧪) 또는 내부망 신뢰 + 서비스 토큰. 스트림 키는 채널당 회전 가능 시크릿.

## 6. 데이터 모델 & Proto 설계 (초안 §1단계 구체화)

### 6.1 관계형 스키마 (Postgres, 개략)

- `users(id, email UNIQUE, password_hash, display_name, created_at)`
- `channels(id, owner_user_id FK, slug UNIQUE, title, category, stream_key_hash, created_at)`
- `streams(id, channel_id FK, started_at, ended_at NULL, peak_viewers, status)` ← 방송 세션 이력
- `follows(follower_user_id FK, channel_id FK, created_at, PK(follower,channel))`
- (선택) `chat_messages_archive(...)` — 실시간은 Redis, 영구보관은 배치 적재.

> 규칙: 모든 조회는 **prepared statement 바인딩** + **ORDER BY 명시**. Drizzle 마이그레이션으로 스키마 버전관리.

### 6.2 Proto 규격 (서비스 계약, buf 관리)

```
proto/
  user/v1/user.proto        # User, AuthService (Login, Refresh, Me)
  channel/v1/channel.proto  # Channel, ChannelService (Create, Get, ListLive, StartStream, StopStream, ValidateStreamKey)
  chat/v1/chat.proto        # ChatMessage, ChatService (Join(stream), Send, Moderate)  ← 서버 스트리밍 예시
  common/v1/common.proto    # 공통 타입(Pagination, Timestamp 래퍼, Error 상세)
```

- 버전 디렉터리(`v1`)로 진화 관리. `buf breaking`으로 파괴적 변경 차단.
- Zod 스키마는 proto와 **중복이 아니라 경계 검증용**(브라우저 폼·외부 입력). 내부 gRPC는 proto 타입 신뢰.

## 8. 단계별 작업 계획 (WBS) — 산출물 + 완료조건(DoD)

> 초안의 5단계(8주)를 **Phase 0(부트스트랩) 추가 + 각 단계 산출물·DoD 명시**로 재구성.
> 원 일정(1~~8주차)을 유지하되 Phase 0을 1주차 앞단에 흡수한다. **일정 성립 전제는 §1.4**(4~~5명 + 미디어 스파이크 선행).

**DoD 게이트 규약(중요)**: 각 DoD는 "기록/측정"이 아니라 **pass/fail 판정**이어야 한다. 성능 항목은 "§1.3 목표 충족 **또는** 수용된 편차를 근거와 함께 문서화"를 통과 조건으로 한다. _(리뷰 반영: 측정≠게이트)_

**실행 철학(v1.2, 중요)**: **완료 판정의 기준은 "테스트가 통과했다"가 아니라 "기능이 실제로 구동/배포되어 동작한다"**이다. 각 Phase DoD는 가능한 한 **실 구동 스모크(실제 curl/grpcurl/OBS/브라우저 조작 또는 배포 후 헬스)** 로 닫는다. 자동 테스트는 **크리티컬 패스 회귀 방어에만 얇게** 작성하고(커버리지 목표 없음), 해피패스 E2E는 배포 게이트용 1개만 유지한다. 나머지 검증은 **실제 앱을 돌려서 눈으로/스모크로 확인**한다. _(테스트 최소화·구현/배포 최대화)_

### 8.0 환경 & 선행 스파이크 (Phase 0 병행) _(리뷰 반영: 성능 DoD의 측정 환경 부재 보완)_

- **환경 토폴로지**: `dev`(로컬 docker-compose) / `stage`(성능·부하 측정 + 배포 리허설 — **하이브리드 프리뷰: Vercel 프리뷰 + Fly.io stage 앱**, §1.3 수치의 기준 환경) / **`prod`(실 배포 대상 — ADR-8 하이브리드: Vercel + Fly.io + R2)**. stage/prod는 채팅 팬아웃·미디어 지연 측정을 위해 노트북이 아닌 실 호스트(Fly 지정 사양 VM)로 확보하며, **§1.3 성능·배포 DoD는 이 배포된 환경에서 측정한다**. _(v1.2: prod를 후속→실 배포 대상으로 승격)_
- **k6 WebSocket 부하 리그**: 1,000+ 가상 시청자가 한 채널에 접속·수신하는 시나리오 스크립트를 **작업 아이템으로 선(先)작성**(§1.3 팬아웃 수치 측정 도구).
- **glass-to-glass 측정법**: 송출 화면에 가시 타임코드를 띄우고 재생 프레임과 대조하는 절차 문서화.
- **미디어 스파이크(필수 선행)**: `Node-Media-Server`의 실 LL-HLS 지연을 stage에서 실측 → 목표 미달 시 `MediaMTX`/SRS 전환 결정(ADR-3의 필수 게이트). Phase 3 착수 전 완료.

- [ ] **미디어 → Fly.io prod**: RTMP **TCP 1935 전용 IP** 개방, LL-HLS 출력 → **R2 업로드**(S3 호환 클라이언트), 서명 URL 도메인=R2/CDN 도메인으로 발급
- [ ] **데이터**: Fly Postgres(또는 매니지드) + Upstash/Fly Redis 프로비저닝, **prod 마이그레이션 실행**, 시드(데모 채널) 적재
- [ ] **크로스오리진 마감(§10)**: Vercel↔Fly CORS 허용 오리진, WS `Origin` 검증, refresh 쿠키 `SameSite`/도메인, 서명 URL CORS 확정
- [ ] **시크릿 분산 관리**: JWT 시크릿·스트림키 솔트·R2 키를 Vercel env + Fly secrets에 분리 주입(§10)
- [ ] **CD 자동화**: `main` 머지 → GitHub Actions가 Vercel deploy + `flyctl deploy` 실행, **배포 후 스모크(실 도메인 `/health`·재생·채팅)**, 실패 시 **이전 릴리스로 롤백**(`flyctl releases`·Vercel rollback)
- **산출물**: 공개 URL에서 접속 가능한 라이브 서비스, `main` 자동 배포 파이프라인, 롤백 절차 문서.
- **DoD(배포 게이트, pass/fail)**: **실 도메인에서 F1~F6 스모크 전부 동작**(회원가입→방송→시청→채팅), OBS→실 도메인 재생 성공, **무단 URL 재생 차단(prod)**, `main` 머지 1회로 web+백엔드+미디어가 **무중단에 가깝게 자동 배포**되고 배포후 스모크 그린, 강제 롤백 1회 리허설 성공.

> **일정 버퍼 주의**: Phase 0~6가 8주에 꽉 차 있고 Phase 3(미디어)·Phase 6(배포)이 초과 리스크 상위다. §1.4 조건(인원/스파이크 선행)을 못 지키면 **일정 재산정**이 전제다. 배포를 Phase 6로 몰지 말고 **Phase 0의 파이프라인 뼈대 + Phase 3부터 미디어 실배포 리허설**로 위험을 앞당겨 분산한다. _(v1.2: 버퍼·배포 리스크 반영)_

## 10. 비기능 요구사항 (NFR)

- **성능**: §1.3 목표치. 목록은 Redis 캐시 우선, 채팅은 룸 샤딩 여지 확보.
- **확장성**: 서비스별 수평 확장. 채팅 팬아웃은 Redis Pub/Sub→(필요시)Streams/전용 브로커. 미디어는 CDN 오프로딩.
- **가용성 & Redis SPOF 대응**: Redis가 라이브상태·채팅·목록캐시 3개 평면을 겸하는 SPOF임을 인지. 구체 폴백 — **Redis 장애 시 `ListLive`는 Postgres `streams.status`로 폴백**, **채팅은 unavailable로 표시하되 앱 크래시 금지**. HA 해법은 Redis Sentinel/Cluster(🧪 후속). 헬스체크·리트라이·서킷브레이커 공통 적용. _(리뷰 반영: SPOF 폴백 구체화)_
- **비용/보존(video 특성)**: CDN egress + 오브젝트 스토리지 증가가 비용 핵심. dev는 MinIO, **prod는 Cloudflare R2**(egress 무료 → 비디오 비용 리스크 완화, ADR-8). dev/prod 모두 **S3 호환 인터페이스로 코드 동일**. HLS 세그먼트·썸네일은 **retention TTL로 정리**(무한 증가 금지). Fly VM·대역폭·Postgres/Redis 매니지드 비용은 prod 규모에 따라 모니터링. _(v1.2: 실 CDN=R2로 구체화)_
- **배포·크로스오리진(ADR-8, v1.2)**: web(Vercel)과 백엔드(Fly)가 **다른 오리진**이므로 — (a) BFF **CORS 허용 오리진**을 Vercel 도메인(+프리뷰 와일드카드)으로 명시, (b) **WS 핸드셰이크 `Origin` 검증**, (c) refresh 쿠키는 크로스사이트 대응(`SameSite=None; Secure` + 도메인 전략, 또는 BFF를 web과 동일 상위도메인 서브도메인으로 배치), (d) **서명 HLS URL의 CORS**(R2 버킷 CORS에 Vercel 오리진 허용)까지 일괄 확정. 콜드스타트 방지 위해 BFF·미디어 **min-instances ≥ 1**.
- **보안·시크릿(크로스플랫폼)**: argon2 해시, JWT 회전, WS 인증, 입력 검증(Zod), 레이트리밋(§8 Phase 2), 스트림 키 비노출·회전, 서명 재생 URL, 의존성 감사(`pnpm audit`). **시크릿은 `.env`(dev)→Vercel env + Fly secrets(prod)로 분리 주입**하며 저장소에 커밋 금지. JWT 시크릿/스트림키 솔트/R2 액세스키는 백엔드(Fly) 전용, web(Vercel)에는 퍼블릭 URL만 노출.
- **개인정보**: MVP는 채팅 로그 영구 아카이브 없음(Redis 실시간만) → GDPR/보존 표면 최소화. 아카이브 도입 시 보존기간·삭제요청 처리 별도 설계.
- **관측성(MVP 스코프)**: 구조화 로그(pino) + 기본 메트릭(Prometheus)까지. **분산추적(OTel)·대시보드(Grafana)·중앙로그(Loki)는 후속**으로 이월(§1.2). _(리뷰 반영: 8주 범위 과대 방지)_
- **DX**: 모노레포 캐시(Turbo), 로컬 원커맨드 기동(compose), 타입 공유(proto/schemas).

## 14. 즉시 착수 체크리스트 (Phase 0 첫날)

1. `pnpm dlx create-turbo@latest` 기반 스캐폴딩 → `apps/*`, `packages/*` 골격.
2. `infra/docker-compose.yml`에 postgres/redis/minio/nginx 정의 후 `docker compose up -d`.
3. `buf.yaml`/`buf.gen.yaml` 작성, `packages/proto`에 `common/v1` 넣고 `buf generate` 확인.
4. GitHub Actions 워크플로(lint/typecheck/buf/build) 추가, PR 규칙 설정.
5. `/health` 엔드포인트로 BFF↔Postgres↔Redis 연결 스모크.
6. **배포 뼈대(ADR-8)**: `apps/web` Vercel 프로젝트 링크(`vercel link`), 백엔드/미디어 `Dockerfile`+`fly.toml`(`flyctl launch`), R2 버킷 생성, **빈 "Hello" 서비스를 Vercel·Fly에 1회 실배포**해 파이프라인 생존 확인.

> 착수 직전, 변동성 높은 스택(Connect, Next.js, buf 플러그인, 미디어 서버, **flyctl·Vercel·R2 설정**)은
> **context7 MCP로 최신 공식 문서를 확인**하고 버전/설정을 고정한다.
