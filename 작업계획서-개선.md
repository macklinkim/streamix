# [개선 작업계획서] 미디어 인코딩 최적화 & 실시간 팬아웃 견고화

> **v1.0 (2026-07)** · 본 문서는 [`작업계획서.md`](./작업계획서.md)(이하 "본계획서")의 **후속 개선 계획**이다.
> MVP에서 "수용된 편차"로 기록한 항목(g2g 지연)과 §1.2 비범위였던 항목 중 일부(LL-HLS)를 재개방하고,
> 코드 실사에서 발견된 인코딩 결함을 작업화한다. 본계획서의 규약(ADR 번호 연속, DoD pass/fail 게이트,
> 구동 스모크 1순위 — §9)을 그대로 따른다.

## 0. 이 문서를 읽는 법

- **범위**: `svc-media`(인코딩·패키징)와 채팅 팬아웃 경로(`bff/ws/chat.ts`, `svc-media/ingest.ts`)만 다룬다. 제어 평면(Core/인증/DB)은 불변.
- **전제**: 본계획서 Phase 3~6 완료 상태(현 `main`). 모든 성능 수치는 본계획서 §8.0의 stage 환경 + 명시된 측정법으로 판정한다.
- **우선순위**: M0(측정) → M1(무비용 수정) → M2(인제스트 remux) → M3(LL-HLS) → M4(팬아웃 견고화). M5는 선택.
- **실행 구조**: 병렬 서브에이전트 트랙 구성·파일 소유권·머지 순서는 [`KICKOFF.md`](./KICKOFF.md) — 트랙 배치가 본 문서 §4 순서와 다르면 KICKOFF가 우선.

## 1. 배경 — 현재 상태 (코드 실사 기준)

### 1.1 인코딩 파이프라인 사실

| 경로          | 현재 구현                                                                                     | 위치                           |
| ------------- | --------------------------------------------------------------------------------------------- | ------------------------------ |
| OBS 송출      | RTMP → NMS → ffmpeg `-c copy`(재인코딩 없음) → 표준 HLS(`hls_time=2`, `list_size=4`, TS)      | `apps/svc-media/src/main.ts`   |
| 브라우저 송출 | MediaRecorder(webm) → WS → ffmpeg **풀 트랜스코딩**(libx264 veryfast + aac 44.1k) → 로컬 RTMP | `apps/svc-media/src/ingest.ts` |
| 썸네일        | 주기(15s)마다 ffmpeg 신규 spawn + RTMP 신규 연결로 1프레임 캡처                               | `apps/svc-media/src/main.ts`   |
| 채팅 팬아웃   | svc-chat publish → Redis Pub/Sub → BFF 룸별 refcount 구독 → 소켓 루프 `send`                  | `apps/bff/src/ws/chat.ts`      |

### 1.2 발견된 결함/편차 (작업 동기)

1. **g2g 지연 6~10s** — 본계획서 Phase 3 판정에서 "수용된 편차". 목표(<6s)는 진짜 LL-HLS 없이는 미달(ADR-3 게이트 그대로).
2. **브라우저 인제스트 CPU 과다** — 스트리머 1명당 libx264 풀 인코딩. Fly.io shared CPU에서 최대 비용 항목.
3. **레이트 컨트롤 부재** — 인제스트 트랜스코딩에 `-b:v`/`-maxrate`/`-bufsize` 없음 → libx264 기본 CRF 23 무제한. 장면 복잡도에 따라 비트레이트 폭주 → 시청자 버퍼링.
4. **GOP 하드코딩/미정렬** — `-g 60`은 30fps 가정. OBS 경로는 `-c copy`라 세그먼트 분할이 소스 키프레임 간격에 종속인데, 키프레임 간격 요구사항(2s)이 어디에도 문서화되어 있지 않음.
5. **불필요 리샘플** — MediaRecorder 오디오는 Opus 48kHz인데 `-ar 44100`으로 다운리샘플.
6. **backpressure 부재** — ① BFF 팬아웃이 `socket.bufferedAmount` 무시하고 무조건 `send`, ② 인제스트 WS가 `sink.write()` 반환값 무시(ffmpeg가 밀리면 Node 힙에 무한 버퍼링).

### 1.3 사실 검증 메모 (외부 의존 결정의 근거)

- **FFmpeg hls muxer는 진짜 LL-HLS 불가** — `#EXT-X-PART`/`#EXT-X-PRELOAD-HINT`/blocking playlist reload 미지원(6/7/master 동일). dash muxer의 `-lhls`는 폐기된 커뮤니티 `#EXT-X-PREFETCH`만 출력("This is not Apple's version LHLS"). → LL-HLS는 패키저 교체로만 가능.
- **SRS는 LL-HLS 미구현** — 본계획서 ADR-3의 "MediaMTX/SRS" 중 SRS는 후보 탈락(공식 문서가 저지연은 HTTP-FLV/SRT/WebRTC 권고). Shaka Packager도 LL-DASH만. 유효 후보: **MediaMTX**(LL-HLS 기본 활성, RTMP 인제스트), **OvenMediaEngine**(v0.14+).
- **브라우저 H.264 캡처 지원** — Chrome 126+/Safari: `video/mp4;codecs=avc1.42E01E,mp4a.40.2`(fMP4, H.264+AAC) → 비디오·오디오 모두 copy remux 가능. 구형 Chromium: `video/webm;codecs=h264`(비디오 copy, 오디오 Opus→AAC 변환 필요 — FLV가 Opus 미수용). Firefox: VP8만 → 풀 트랜스코딩 폴백 유지.

## 3. 아키텍처 결정 기록 (ADR) — 본계획서 ADR-8에서 연속

### ADR-9. 브라우저 인제스트: **코덱 협상 + copy remux** (트랜스코딩은 폴백) ✅

- **결정**: 스튜디오 페이지가 `MediaRecorder.isTypeSupported`로 3단 협상 → 선택 코덱을 WS 쿼리 파라미터로 전달 → `ingest.ts`가 코덱별 ffmpeg 인자 분기.
  1. `video/mp4;codecs=avc1.42E01E,mp4a.40.2` → `-c:v copy -c:a copy` (인코딩 0)
  2. `video/webm;codecs=h264` → `-c:v copy -c:a aac -ar 48000` (오디오만 변환)
  3. 그 외(VP8 등) → 현행 풀 트랜스코딩 + M1의 레이트 컨트롤 적용
- **근거**: §1.3 검증 메모. 인코딩 제거가 서버 CPU 최대 절감이며, 화질 열화(재인코딩 세대 손실)도 제거.
- **주의**: fMP4 스트리밍 입력 시 ffmpeg에 `-f mp4` 명시 대신 파이프 자동 감지 실측 필요(스파이크 항목). Android는 HW 인코더 유무에 따라 지원 편차 → 런타임 feature-detect 필수.

### ADR-10. LL-HLS 패키저: **MediaMTX 채택** (NMS+ffmpeg 패키징 대체) ✅

- **결정**: RTMP 인제스트~LL-HLS 패키징을 MediaMTX로 교체. NMS·ffmpeg HLS 패키징 프로세스 제거. `hls-server.ts`의 서명 URL authz는 **MediaMTX 앞단 프록시**로 유지(서명 검증 후 리버스 프록시).
- **근거**: ffmpeg 단독 LL-HLS 불가(§1.3). SRS·Shaka 탈락. MediaMTX는 LL-HLS 기본 활성 + RTMP 인제스트 + 단일 바이너리(Fly.io 배포 단순). 대안 OvenMediaEngine은 기능 동급이나 운영 표면적 큼 — 스파이크에서 MediaMTX 미달 시에만 검토.
- **연쇄 변경**: postPublish/donePublish 훅 → MediaMTX `runOnReady`/`runOnNotReady` 훅으로 이관(스트림키 검증 gRPC 호출 유지). 썸네일·retention은 MediaMTX 세그먼트 경로 기준으로 수정.
- **리스크**: 본 개선안 중 유일한 구조 변경. 반드시 스파이크(M3-1) 게이트 뒤에서만 착수.

### ADR-11. **LMAX Disruptor 도입 반려** — 대체: backpressure + 배칭 ❌→✅

- **검토 결과(반려)**: Disruptor의 전제조건 — 멀티스레드 공유메모리 생산자/소비자, 락·CAS 경합, cache-line false sharing, 큐 노드 GC 압력 — 이 **Node 단일 스레드 이벤트 루프에는 전부 부재**. 단일 스레드에서는 동기화 없는 원형 버퍼로 퇴화하여 패턴의 존재 이유가 소멸한다.
- **정량 근거**: 채팅 핫패스 병목은 Redis 왕복 + WS write 시스콜(네트워크 I/O). 목표치(1,000 동접, p95 300ms)는 Disruptor 설계 영역(수백만 msg/s, µs 지연)과 3~4자릿수 괴리. Node 생태계 구현체(`shared-memory-disruptor`)는 사실상 방치 상태로 프로덕션 근거 없음.
- **대체 결정**: 같은 문제 공간(고부하 팬아웃)의 실제 결함을 푼다 — M4의 WS backpressure·메시지 배칭·인제스트 backpressure. Disruptor의 **single-writer principle 자체는 이미 적용 중**(라이브 상태 SET NX 단일 writer, §5.2)이므로 문서/회고에서 원리로 인용한다.
- **재개방 조건**: 팬아웃 허브를 JVM/Go 등 멀티스레드 런타임으로 분리하는 결정이 생길 때만. 그 경우도 본 리포가 아닌 별도 실험 리포에서 벤치마크로 검증 후 판단.

## 6. 검증 전략

본계획서 §9 그대로: 구동 스모크 1순위, 자동 테스트는 크리티컬 패스만 얇게. 본 계획에서 추가되는 얇은 회귀 방어는 두 개뿐 — ① 코덱 협상 분기(유닛: codec 파라미터→ffmpeg 인자), ② 서명 프록시(스모크: 서명 200/무서명 403). 나머지는 전부 실송출 스모크와 stage 실측으로 닫는다.
