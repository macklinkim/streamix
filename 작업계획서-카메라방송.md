# [기능 작업계획서] 카메라 라이브 방송 — "방송시작" 원클릭 Go Live

> **문서 버전**: v1.0 · **작성 기준일**: 2026-07-07
> 본 문서는 `작업계획서.md`(본계획서)와 `작업계획서-개선.md`(개선계획)의 후속 기능 계획이다. 규약을 전부 상속한다: ADR 번호 연속(ADR-12부터), DoD는 pass/fail 게이트, 구동 스모크 1순위, 완료 판정 = 실 구동 + 배포 헬스(테스트 통과 아님).
> **실행 방식**: 본 문서는 `/loop`(동적 모드)로 **완성까지 무인 자동 실행**된다. §0.1 무블로킹 규약이 다른 모든 규약보다 우선한다.

## 0. 이 문서를 읽는 법 · 실행 규약

- **범위**: 노트북/핸드폰 **카메라**로 브라우저에서 바로 라이브 송출(방송시작 버튼 → 카메라 → 시청자 재생). 기존 화면 송출·OBS RTMP 경로는 유지·재사용.
- **전제**: 미디어 평면(WS `/ingest`→ffmpeg→MediaMTX→LL-HLS 서명 재생)은 prod 가동 중. 이 계획은 **인제스트 파이프라인을 바꾸지 않는다** — 브라우저 캡처 소스와 Go-Live UX를 추가한다.
- **우선순위**: M1 → M2 → M3 → M4 순서 고정. 앞 마일스톤의 게이트 실패가 뒤를 막으면 원인 수정이 우선, 외부 블로커면 보류 후 진행.

### 0.1 무블로킹 자율 루프 규약 (최우선)

1. **사용자 입력 대기 금지.** 어떤 항목도 사용자 응답을 기다리며 정지하지 않는다. 질문이 생기면 (a) 본 문서의 ADR·가정에 따라 스스로 결정하고 결정 내용을 해당 ADR에 append, (b) 결정 불가(자격증명·실기기·결제·계정 동의 필요)면 해당 항목을 `[~]` 보류로 표시하고 §8 보류 레지스터에 기록한 뒤 **즉시 다음 항목으로 진행**한다.
2. **보류 허용 사유(이것만)**: 실기기(아이폰/안드로이드) 검증, 시크릿/자격증명 발급, 결제 리소스, 사용자 계정의 OAuth 동의. 그 외 "애매해서 보류"는 금지 — 결정하고 전진.
3. **DoD에 "사용자 확인" 류 문구 금지.** 모든 판정은 에이전트가 스스로 실행 가능한 스모크·스크립트·헤드리스 브라우저(fake device)로 닫혀야 한다.
4. **리소스 배분: 구현·조사 90% / 검증 10%.** 신규 테스트 인프라 작성 금지. 마일스톤당 얇은 스모크 **최대 1개**(기존 스모크 스크립트 패턴 재사용) + 기존 스모크 회귀만. 유닛 테스트 신규 작성 금지. 커버리지 무시.
5. **마일스톤 단위 커밋.** 각 마일스톤 완료 시 conventional commit(소문자 subject) + 본 문서 체크박스 갱신 + 메모리(`streamix-loop-state.md`) 갱신. 배포는 M4에서 일괄.
6. **루프 종료 조건**: C1~C4 전부 판정 완료(PASS 또는 편차 수용) 또는 남은 항목이 전부 보류.

## 3. ADR (ADR-11에서 연속)

### ADR-12. 카메라 인제스트 경로: **기존 MediaRecorder→WS 파이프라인 재사용** ✅

- **결정**: 카메라도 `getUserMedia` → MediaRecorder → WS `/ingest` 기존 경로로 송출. WebRTC/WHIP 신규 도입은 하지 않는다.
- **근거**: 파이프라인이 prod 검증 완료(코덱 협상·backpressure·키검증·라이프사이클 전부 재사용). MediaMTX가 WHIP을 지원하지만 Fly에서 WebRTC는 UDP/ICE 포트·전용 IP 구성이 필요해 인프라 리스크가 크고, 현 지연(1.3~2.2s)은 수용 범위.
- **대안**: WHIP(sub-second 지연, 모바일 코덱 문제 원천 회피) — 기각.
- **재개방 조건**: MediaRecorder 기반 모바일 송출이 게이트 C3를 에뮬레이션에서도 못 넘거나, 지연 요구가 <1s로 강화될 때.

### ADR-13. Go-Live 인증: **단기 브라우저 인제스트 토큰** (OBS 키와 분리) ✅

- **결정**: `ChannelService.IssueIngestToken`(브라우저 노출, 소유자 인증) 신설. Core가 `bit_` prefix 불투명 토큰을 Redis `ingest:{token}`→channelId 매핑(TTL 15m)으로 발급. svc-media는 무변경 — 기존 `ValidateStreamKey`가 `bit_` prefix면 토큰 경로로 조회해 동일 응답 반환.
- **근거**: 사실 4의 rotate 풋건 제거가 "방송시작 버튼" UX의 전제. 검증은 WS 연결 시 1회이므로 짧은 TTL로 충분(재연결 시 웹이 재발급).
- **대안**: (a) rotate 유지 — OBS 파괴 풋건 존속, 기각. (b) plaintext 키 서버 보관 — 해시 저장 설계 후퇴, 기각.
- **연쇄 변경**: proto 추가 → **BFF 재배포 필수**(unknown field 드랍 교훈, Polish 2차).

### ADR-14. 방송 중 카메라 전환: **정지 후 자동 재시작** ✅

- **결정**: MediaRecorder는 트랙 교체 미지원 → 전후면/장치 전환 = 현재 세션 정상 종료 후 새 ingest 연결로 자동 재시작. 시청자 측 수 초 공백은 **수용 편차**(watch 폴링이 오프라인↔라이브 자동 전환하므로 회복은 자동).
- **대안**: canvas 합성으로 무중단 전환 — CPU·복잡도 대비 가치 낮음, 기각.

## 4. WBS

> 각 마일스톤 = 독립 커밋 단위. 체크박스는 루프가 갱신. 검증은 §0.1-4 규약(마일스톤당 스모크 최대 1개).

### M0 — 조사 스파이크 (~0.5d)

- [x] S1: fake-device 카메라 캡처 확인 — Chromium은 기본 `mp4-h264` 우선 지원하나 **fMP4·webm+h264 copy 둘 다 파이프 디먹스 실패**. `webm-vp8` transcode만 안정 → 클라 코덱 우선순위 vp8로 변경(§2.1 발견). C1 측정기 성립 확인 완료.
- [x] S2: 세로 영상 통과 — 별도 testsrc 대신 M3 구현으로 해소: transcode 경로는 임의 해상도 통과(파이프라인 스케일 필터 불필요), 세로 letterbox는 클라 `object-contain`로 처리(watch·미리보기). C3 모바일(Pixel 5 세로) prod PASS로 실증.
- **산출물**: 결과를 §2.1·§8 기록(코드 산출물 없음).
- **DoD**: S1·S2 결과 기록 완료.

### M1 — 데스크톱 카메라 방송 (ADR-12, ~1d)

- [x] `broadcast.tsx` 개편: 소스 선택 `camera`(기본) | `screen`. camera = `getUserMedia({video:{width:1280,height:720,frameRate:30,deviceId?}, audio:{echoCancellation,noiseSuppression,deviceId?}})`. 이후 MediaRecorder→WS 경로는 기존 코드 공유.
- [x] 장치 선택: `enumerateDevices` 카메라/마이크 셀렉트(방송 시작 전). 권한 거부(NotAllowedError 등) 에러 메시지 표시 — 현재 침묵 리턴 버그 동시 수정.
- [x] `new MediaRecorder` try/catch(미지원 구성 시 에러 상태 전이 — 기존 잠복 버그 수정).
- [x] 스튜디오 UI: "방송시작" 카드 — 소스·장치 선택, 카메라 미리보기(muted), 시작/종료, LIVE 표시. 기존 ScreenBroadcast와 단일 컴포넌트로 통합(`BroadcastPanel`).
- [x] 검증: C1 해피패스 스크립트 1개(`apps/web/e2e/camera-broadcast.mts`, 기존 `broadcast-prod.mts` 패턴 복제·로컬 타깃, SOURCE/MOBILE env 파라미터).
- **산출물**: 통합 BroadcastPanel, 스튜디오 Go-Live 카드, C1 스크립트.
- **DoD**: **C1 통과**(로컬 풀스택). 기존 화면송출 시작·종료 동작 유지(같은 스크립트로 소스=screen 1회 실행). — 코드 완료, tsc 그린. 풀스택 실행 판정은 M2(자동 토큰 UX) 뒤 C1+C2 합동 실행에서.

### M2 — 원클릭 방송시작: 인제스트 토큰 (ADR-13, ~1d)

- [x] proto: `IssueIngestToken`(empty req → `token`, `expires_at`) 추가, buf 코드젠.
- [x] svc-core: 토큰 발급(Redis `ingest:{token}` TTL 15m) + `validateStreamKey`/`startStream` 공용 `resolveChannelId`의 `bit_` prefix 분기.
- [x] BFF: `issueIngestToken` 브라우저 노출(requireUser). internal-only 목록은 불변.
- [x] web: BroadcastPanel이 go-live마다 자동 토큰 발급 → 리로드 후에도 즉시 방송. 스트림 키 패널은 "OBS 전용" 표기로 분리.
- [x] 검증: 스모크 1개(`apps/svc-media/scripts/smoke-ingest-token.ts` — 발급→go-live, 위조 4403, rotate 독립성). 경로는 ffmpeg/ws 의존으로 svc-media/scripts에 배치(계획의 bff/scripts에서 이동).
- **산출물**: proto/core/bff 토큰 경로, 자동 발급 UX, 스모크 1개.
- **DoD**: **C2 코어 로직 PASS**(라이브 core로 검증: bit_ 발급·미래만료·OBS키 불변·채널검증·위조무효·rotate 독립성 R5 전부 통과). svc-media 코드 무변경 확인 ✅(resolver는 core에만). WS-edge 4403 + go-live 실 파이프라인 검증은 M4 prod에서 C1/smoke-ingest-prod와 합동(로컬 MediaMTX 미설치).

### M3 — 모바일(폰) 방송 (ADR-12/14, ~1d)

- [x] facingMode: 시작 전 전/후면 선택(모바일) + 방송 중 전환 = 자동 재시작(`switchCamera`, ADR-14). 전환 UI는 카메라 소스일 때만.
- [x] 세로 화면: 모바일 해상도 강제 제거(`facingMode`만, 브라우저 기본), 미리보기 `object-contain max-h-70vh`(고정 aspect-video 제거), watch 플레이어 `object-contain` letterbox 추가.
- [x] 송출 유지: `navigator.wakeLock` 요청(미지원 무시, 포그라운드 복귀 재요청), `visibilitychange` 백그라운드 경고 배너, `pagehide` 정상 종료.
- [x] 스튜디오 모바일 레이아웃 점검 — 기존 `max-w-lg`는 390px에서 풀폭, 컨트롤 반응형(grid-cols-1) 유지, 수정 불필요.
- [x] 검증: C1 스크립트 모바일 프로파일 파라미터(`MOBILE=1` → Pixel 5 디바이스 뷰포트 + fake video) — M1에서 이미 추가. 실행 판정은 M4 prod에서.
- [~] 실기기(iPhone Safari·Android Chrome) 검증 — **보류(블록 안 함)**: §8 기록 완료. iOS Safari MediaRecorder fMP4 편차(R1)는 실기기에서만 확정.
- **산출물**: 모바일 Go-Live UX(전환·세로·wake lock·배경 배너), 모바일 프로파일 판정.
- **DoD**: **C3 코드 완료**(tsc 그린). 에뮬레이션 go-live 판정은 M4 prod에서 C1 MOBILE=1로 합동 실행(로컬 MediaMTX 미설치). 실기기는 보류로 게이트 제외.

### M4 — 방송 정보 + 가드 + prod 배포 (~1d)

- [x] proto: `UpdateChannel`(title, category — 소유자만) end-to-end(core·BFF 노출·코드젠). 방송시작 카드에 제목 입력(선택, 미입력 시 기존 제목 유지). 라이브 core 검증: 갱신·빈값유지 PASS.
- [x] 중복 ingest 가드: svc-media `activeChannels` set — 동일 채널 두 번째 WS `4409` 즉시 close(사실 7 차단). 가드는 WS close에 해제 → ADR-14 재시작(선종료→후재연결)은 신규 연결로 통과. 브라우저 4409 메시지 추가.
- [x] 배포: svc-core·bff·svc-media Fly 재배포(proto 변경 → BFF 필수) + web Vercel prod 완료(전부 healthy).
- [x] 검증: C1 스크립트 prod 타깃 PASS(camera/mobile/screen 전부 live+세그먼트+오프라인) + `smoke-ingest-prod` 회귀 ALL PASS.
- **산출물**: UpdateChannel, 제목 설정 UX, 중복 가드, prod 반영.
- **DoD**: **C4 통과** + 회귀 그린. §2 판정 결과 표 기입 완료, 메모리 갱신 완료.
