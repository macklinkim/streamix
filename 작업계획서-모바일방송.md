## 0. 이 문서를 읽는 법 · 실행 규약

- **범위**: 핸드폰에서 방송할 때 송출 소스 3종을 지원한다 — (a) **폰 화면**, (b) **폰 카메라(전/후면)**, (c) **폰에 연결된 외부 카메라 장비(GoPro, USB 웹캠 등)**.
- **전제**: 미디어 평면(WS `/ingest`→ffmpeg→MediaMTX→LL-HLS)과 카메라 방송(전/후면 전환, 세로 화면, wake lock, 인제스트 토큰)은 prod 가동 중. 이 계획은 **인제스트 파이프라인을 바꾸지 않는다** — 소스 획득 경로와 스튜디오 UX만 확장한다.
- **우선순위**: M0 → M1 → M2 → M3 → M4 순서 고정.
- **무블로킹 규약**: `작업계획서-카메라방송.md` §0.1 전문을 그대로 상속한다(사용자 입력 대기 금지, 보류 허용 사유 4종, 마일스톤당 신규 스모크 최대 1개, 마일스톤 단위 커밋, 루프 종료 조건).

## 3. ADR (ADR-14에서 연속)

### ADR-15. 폰 화면 송출: **이중 능력 감지 + RTMP 폴백 안내** (네이티브 앱 기각)

- **결정**: screen 소스 게이팅을 이중화한다 — (1) `typeof navigator.mediaDevices?.getDisplayMedia === "function"` 존재 감지로 UI 노출을 제어하고, (2) 호출 실패 경로에서 "사용자 취소"와 "플랫폼 미지원"(API가 존재하지만 항상 거부하는 브라우저 이력, 사실 2)을 구분해 미지원이면 RTMP 폴백 카드로 유도한다. 미지원 시 screen 소스 비활성 + "화면 송출 앱(RTMP)으로 방송하기" 안내 — 서드파티 화면 송출 앱에 붙일 RTMP URL + 스트림 키를 제공한다.
- **근거**: 모바일 브라우저에 화면 캡처 API가 없다(사실 2). 존재 감지 단독은 Chrome Android의 "노출-but-거부" 이력 때문에 불충분. 네이티브 앱(MediaProjection/ReplayKit)은 앱 배포·심사·별도 코드베이스 비용이 크고 비범위. RTMP 인제스트는 prod 검증 완료라 폴백의 서버 비용 0.
- **대안**: (a) 네이티브 앱 — 기각(비용). (b) Android PWA + 실험 API 대기 — 기각(표준화 불투명).
- **재개방 조건**: 모바일 브라우저에 `getDisplayMedia`가 실제 출하되거나, 네이티브 앱 트랙이 별도 승인될 때.

### ADR-16. 외부 카메라(GoPro 등): **폰에서는 RTMP가 주 경로, UVC(getUserMedia)는 iPad/데스크톱 보조 경로**

- **결정**: (1) **주 경로 — 장비 자체 RTMP**: RTMP 송출 지원 장비(GoPro HERO7 Black+, 짐벌 캠 등)는 기존 durable 스트림 키 RTMP 경로로 직결 — 서버 무변경. 폰 사용자의 외부 장비 연동은 이 경로가 **유일한 실동선**이다(사실 3). (2) **보조 경로 — USB UVC → getUserMedia deviceId**: Android Chrome·iPhone Safari는 외부 UVC를 열거하지 않으므로 폰에서는 성립하지 않는다. iPadOS 17+ Safari와 데스크톱에서만 유효하며, M1의 deviceId 선택 UI가 이를 자연 커버한다(별도 작업 없음).
- **근거**: 두 경로 모두 prod 검증된 인제스트 재사용. WHIP/SRT 신설은 ADR-12 기각 사유(Fly UDP/ICE 인프라 리스크) 유효, 지연 수용 범위(1.3~2.2s) 내.
- **대안**: SRT 인제스트 활성화(GoPro 미지원, 장비 호환 협소) — 기각. GoPro 전용 SDK/BLE 제어 — 비범위, 기각. Android Chrome UVC 출하 대기 — 장기 미해결 제약(사실 3), 기각.
- **주의**: 실장비(GoPro USB 웹캠 모드) 확인은 보류 레지스터 — 폰에서는 **실패가 예상 결과**이며, iPad가 있으면 UVC 열거 확인 가치 있음.

### ADR-17. 모바일 카메라 선택 UX: **facingMode 프리셋 + "카메라 더 보기" 토글로 deviceId 목록 노출**

- **결정**: 모바일 UI는 기본 "전면/후면" 프리셋을 유지하고, **"카메라 더 보기" 토글(상시 제공)**로 `enumerateDevices` 기반 상세 셀렉트를 펼친다. deviceId 선택 시 `facingMode` 대신 `deviceId: { exact }` 제약을 사용한다. 방송 중 장치 변경은 ADR-14(정지 후 자동 재시작)를 따른다.
- **근거**: 상세 선택의 실사용 가치는 "외부 장치 감지"가 아니라 **멀티 렌즈(광각/망원)·논리 카메라 상세 선택 + iPad UVC 커버**다(사실 3 — 폰에서 외부 UVC는 애초에 열거 안 됨). videoinput 개수 임계값(≥3) 휴리스틱은 멀티 렌즈 폰에서 항상 발화하고 내장 1개 기기+외장(iPad) 조합은 놓치는 등 근거가 성립하지 않아 **상시 토글로 대체**(v1.1, 아키텍처·모바일 검증 반영).
- **대안**: 항상 전체 목록 노출 — 모바일 기본 UX 후퇴, 기각. `videoinput ≥ 3` 자동 임계값 — 오탐·미탐 양방향 결함, 기각. `getCapabilities().facingMode` 필터 — 브라우저 편차, 기각.

## 4. WBS

> 각 마일스톤 = 독립 커밋 단위. 체크박스는 루프가 갱신. 마일스톤당 신규 스모크 최대 1개.

### M0 — 조사 스파이크 (~0.5d)

- [x] S1: Playwright fake video 장치 **복수 주입** — `--use-fake-device-for-media-stream=device-count=N` 옵션 경로를 1순위로 시도(Chromium FakeVideoCaptureDeviceFactory 파라미터, 공식 문서 빈약). 실패 시 D1 대체 판정(deviceId 제약 전달 스파이 + 기본 장치 go-live)로 확정.
- [x] S2: 모바일 에뮬레이션 `getDisplayMedia` 상태 확인 — 에뮬레이션은 API가 존재할 수 있으므로 `addInitScript` API 삭제 주입으로 미지원 재현(D3 측정기). "존재하지만 호출 시 거부" 케이스의 실패 경로 분기(취소 vs 미지원 구분)를 설계에 포함(ADR-15).
- [x] S3: D2 측정기 확정 — `perf-g2g-stage.mts`의 RTMP push 패턴 재사용 vs `smoke-ingest-prod`에 RTMP 모드 파라미터 추가 중 택일(신규 스크립트 아님). RTMP 송출 중 브라우저 go-live 상호 배제(4409/kick) 1회 확인을 같은 스모크에 포함할지 결정.
- **산출물**: 결과를 §2.1·§8·ADR append로 기록(코드 산출물 없음).
- **DoD**: S1~~S3 결과 기록 완료, D1~~D3 측정기 확정. — **완료(2026-07-17)**.

**S1 결과 — 성립(1순위 방식 확정)**: `--use-fake-device-for-media-stream=device-count=3` → `enumerateDevices`가 `fake_device_0..2` 3개 노출(플래그 없으면 1개). D1 대체 판정 불필요. 단 fake 장치는 영상이 동일해 화면만으로 선택을 증명하지 못하므로, D1은 `getUserMedia` 제약 스파이(`addInitScript` 래핑)로 `deviceId.exact` 전달을 함께 판정한다.

**S2 결과 — API 존재 확인**: 에뮬레이션(Pixel 5 컨텍스트)에서 `typeof navigator.mediaDevices.getDisplayMedia === "function"`. 따라서 D3 측정기는 `addInitScript`로 `MediaDevices.prototype.getDisplayMedia`를 삭제해 미지원을 재현한다. "존재하지만 거부" 케이스의 감지 규칙은 ADR-15 (2)로 확정: **모바일 UA에서 screen `NotAllowedError`는 취소가 아니라 미지원으로 판정**(폰에는 취소할 피커가 없으므로 거부는 곧 미지원). 데스크톱은 기존 취소 처리 유지.

**S3 결과 — D2 측정기 = `perf-g2g-stage.mts` ffmpeg RTMP push 패턴 재사용**: `smoke-ingest-prod`는 WS 전용이라(사실 8) RTMP 모드 추가는 스크립트 성격을 바꾼다. 상호 배제(4409) 확인은 **D2-a에 미포함으로 결정** — 4409 가드는 svc-media WS `activeChannels` 경로이고 RTMP 선점은 poller 경유라 판정 경로가 달라 같은 스모크에 섞으면 실패 원인이 모호해진다. M2는 안내 문구만 제공하고 검증은 §8 등재.
